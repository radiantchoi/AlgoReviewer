## LeetCode No.98 Validate Binary Search Tree 풀이 코드 리뷰

안녕하세요, 시니어 알고리즘 코드 리뷰어입니다. LeetCode 98번 'Validate Binary Search Tree' 문제 풀이 코드를 상세하게 리뷰해 드리겠습니다.

---

### 1. 시간 복잡도 (Big-O 분석 및 최적화 가능성)

*   **분석:**
    제공된 솔루션은 트리의 모든 노드를 정확히 한 번씩 방문합니다. 각 노드에서 수행하는 작업(값 비교 및 재귀 호출)은 상수 시간 O(1)입니다. 따라서 전체 시간 복잡도는 트리의 노드 수 N에 비례합니다.
*   **결론:** `O(N)`
*   **최적화 가능성:** 이 문제는 모든 노드의 유효성을 검증해야 하므로, 모든 노드를 최소 한 번은 방문해야 합니다. 따라서 `O(N)`보다 더 효율적인 시간 복잡도를 달성하는 것은 불가능합니다. 현재 풀이는 시간 복도 측면에서 최적입니다.

---

### 2. 공간 복잡도

*   **분석:**
    재귀 호출은 함수 호출 스택을 사용합니다. 최악의 경우(예: 한쪽으로 치우친(skewed) 트리, 즉 연결 리스트와 같은 형태) 재귀 깊이는 트리의 높이 H와 같아집니다.
    *   균형 잡힌 트리의 경우 `H = log N`
    *   완전히 치우친 트리의 경우 `H = N`
*   **결론:** `O(H)`, 여기서 H는 트리의 높이입니다.
*   **최적화 가능성:** 재귀를 사용하는 트리 순회 문제에서 `O(H)`의 공간 복잡도는 일반적이며 합리적입니다. Python의 기본 재귀 제한을 초과하는 매우 깊은 트리가 있을 수 있는 환경이 아니라면, 대부분의 경우 문제가 되지 않습니다. (LeetCode 환경에서는 보통 `H`가 Python의 기본 재귀 제한 내에 있도록 테스트 케이스가 구성됩니다.)

---

### 3. 보편적인 "정석" 풀이 방법

제공된 `min/max bound`를 이용한 재귀적 유효성 검사 방법은 이 문제의 가장 **정석적이고 널리 사용되는 접근 방식 중 하나**입니다. 이 방법은 각 서브트리가 유효한 BST 규칙을 따르는지 확인하기 위해 재귀적으로 하위 노드에 유효 범위(최소값 및 최대값)를 전달합니다.

**또 다른 정석적인 접근 방식**으로는 `In-order Traversal`을 활용하는 방법이 있습니다.
유효한 BST를 In-order Traversal(왼쪽 -> 루트 -> 오른쪽)하면 노드 값들이 **엄격하게 오름차순**으로 정렬됩니다. 이를 이용하여, 순회 중에 이전 노드의 값(prev)을 추적하며 현재 노드의 값이 prev보다 작거나 같으면 유효하지 않은 BST로 판단할 수 있습니다.

두 방법 모두 `O(N)` 시간 복잡도와 `O(H)` 공간 복잡도를 가지며, 본 풀이는 BST의 정의를 직접적으로 구현한다는 점에서 직관적이고 강력합니다.

### 4. 현재 풀이의 적절성과 개선 가능한 부분 (엣지 케이스 포함)

*   **적절성:**
    *   현재 풀이는 BST의 핵심 속성(왼쪽 자식은 부모보다 작고, 오른쪽 자식은 부모보다 크며, 이는 전체 서브트리에 걸쳐 적용됨)을 정확하게 모델링하고 있습니다.
    *   `min`과 `max` 매개변수를 사용하여 각 노드가 유효한 범위 내에 있는지 확인하는 방식은 매우 정확하며, 중간 노드가 전체 서브트리에 미치는 영향을 잘 반영합니다.
    *   **엣지 케이스 처리:**
        *   `root is None` (빈 트리): `True`를 반환하여 올바르게 처리됩니다. 빈 트리는 유효한 BST입니다.
        *   단일 노드 트리: `isValidNode(node, -inf, +inf)` 호출 시 `node.val`은 이 범위 내에 있으므로 `True`를 반환합니다. 좌우 자식은 `None`이 되어 재귀적으로 `True`를 반환하므로 정확합니다.
        *   정수 한계 값: LeetCode 문제에서 노드 값의 범위는 `-2^31 <= Node.val <= 2^31 - 1`로 주어집니다.
            *   코드에서 `criteria = 2**31`로 설정하고 초기 호출에서 `-(criteria + 1)`과 `criteria`를 사용한 것은 이러한 제약 조건을 정확하게 다루기 위함입니다.
            *   즉, `-(2**31 + 1)`과 `2**31`을 초기 `minimum` 및 `maximum`으로 사용하여, `-2^31`과 `2^31 - 1`을 포함하는 범위를 허용하면서도, `root.val <= minimum` 또는 `root.val >= maximum` 조건에서 엄격한 부등호(`>`, `<`)를 올바르게 적용할 수 있도록 합니다. 이는 매우 세심하고 정확한 처리입니다.

*   **개선 가능한 부분:**
    *   **초기 경계 값의 명시성:** `criteria = 2**31`과 같이 하드코딩된 정수 값은 문제의 특정 제약 조건에 의존합니다. Python은 임의 정밀도 정수를 지원하므로, `-float('inf')`와 `float('inf')`를 초기 `minimum` 및 `maximum`으로 사용하는 것이 좀 더 일반적이고 플랫폼 제약에 독립적인 코드가 될 수 있습니다. 물론 LeetCode 환경에서는 현재 방식도 완벽히 유효합니다.
    *   변수명 `criteria`는 다소 일반적입니다. `int_max_bound_for_problem` 등 좀 더 구체적인 이름을 사용하면 의도를 명확히 할 수 있습니다. (이는 아주 사소한 개선점입니다.)

*   **클래스 및 메서드 구조:** `Solution` 클래스 안에 `isValidBST`와 헬퍼 함수 `isValidNode`를 둔 구조는 LeetCode 스타일 가이드라인을 잘 따르고 있습니다.
*   **가독성:** 코드가 깔끔하고 논리적 흐름이 명확하여 읽기 쉽습니다. 조건문(`if/elif`)과 재귀 호출이 잘 배치되어 있습니다.
*   **재귀 활용:** 트리를 순회하는 문제에서 재귀는 Python에서 자연스럽고 흔한 접근 방식입니다. 특별히 비표준적이거나 비효율적인 방식은 없습니다.
