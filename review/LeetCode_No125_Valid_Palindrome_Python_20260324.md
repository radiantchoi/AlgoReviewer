# LeetCode No.125 Valid Palindrome Code Review

## LeetCode No.125 Valid Palindrome 코드 리뷰

안녕하세요. Senior Algorithm Code Reviewer로서 제출해주신 `isPalindrome` 풀이 코드를 리뷰하겠습니다. 전반적으로 문제의 요구사항을 잘 이해하고 해결한 깔끔한 코드입니다. 특히 두-포인터(two-pointer) 방식을 활용한 핵심 로직은 매우 적절합니다. 몇 가지 관점에서 더 깊이 살펴보겠습니다.

---

### 1. 시간 복잡도 (Time Complexity)

*   **분석:**
    *   첫 번째 줄 `letters = list(map(lambda x: x.lower(), filter(lambda x: x.isalpha() or x.isnumeric(), s)))` 부분은 다음과 같이 분석할 수 있습니다:
        *   `filter` 함수는 입력 문자열 `s`의 모든 문자를 한 번씩 순회하며 조건을 확인합니다. 이는 `O(N)` 시간(여기서 `N`은 `s`의 길이)이 소요됩니다.
        *   `map` 함수는 `filter`를 통과한 문자들에 대해 `lower()` 함수를 적용합니다. 이 또한 필터링된 문자의 수에 비례하므로 최악의 경우 `O(N)` 시간이 소요됩니다.
        *   `list()`로 변환하는 과정 또한 필터링 및 매핑된 문자의 수에 비례하여 `O(N)` 시간이 소요됩니다.
        *   따라서 전처리 단계의 총 시간 복잡도는 `O(N)`입니다.
    *   `while left < right:` 루프는 `letters` 리스트의 절반가량을 순회합니다. 각 반복에서 상수 시간 `O(1)` 연산(인덱싱, 비교, 증감)이 이루어지므로, 이 부분의 시간 복잡도는 `O(N)`입니다.
*   **결론:** 전체 시간 복잡도는 **O(N)** 입니다.
*   **최적화 가능성:** 이 문제는 모든 문자를 한 번 이상 확인해야 하므로, O(N)보다 더 빠르게 해결할 수는 없습니다. 따라서 현재의 시간 복잡도는 최적입니다.

---

### 2. 공간 복잡도 (Space Complexity)

*   **분석:**
    *   `letters` 라는 새로운 리스트를 생성합니다. 이 리스트는 입력 문자열 `s`에서 알파벳 또는 숫자인 문자들만 담게 됩니다. 최악의 경우 (예: "abcdefg"와 같이 모든 문자가 유효한 경우) `letters` 리스트의 크기는 입력 문자열 `s`의 길이 `N`에 비례합니다.
*   **결론:** 공간 복잡도는 **O(N)** 입니다.
*   **최적화 가능성:**
    *   `O(N)` 공간 복잡도는 이 문제를 해결하는 일반적인 방법 중 하나이지만, **`O(1)` 공간 복잡도로 최적화할 수 있습니다.** 이는 별도의 리스트를 생성하지 않고, 원본 문자열 `s`에 대해 두 개의 포인터를 사용하여 직접 순회하면서 유효한 문자(알파벳, 숫자)를 찾고 비교하는 방식입니다. 이 방식은 후술할 "개선 가능한 부분"에서 예시 코드와 함께 설명하겠습니다.

---

### 3. 보편적인 "정석" 풀이 방법

*   **정석 접근법:**
    1.  **전처리 (Preprocessing):** 입력 문자열에서 유효하지 않은 문자(공백, 구두점 등)를 제거하고, 모든 문자를 소문자 또는 대문자로 통일합니다. 이는 현재 코드의 첫 번째 줄에서 잘 수행되고 있습니다.
    2.  **투-포인터 (Two-Pointers):** 전처리된 문자열(또는 문자 리스트)의 양 끝에서 시작하는 두 개의 포인터를 사용하여 안쪽으로 이동시키면서 문자를 비교합니다. 일치하지 않는 문자가 발견되면 팰린드롬이 아니고, 모든 문자가 일치하면 팰린드롬입니다. 현재 코드의 `while left < right:` 루프가 이 방식을 정확히 구현하고 있습니다.
*   **문제 유형별 적절성:** 이 문제는 동적 계획법(DP)이나 그래프 알고리즘과는 관련이 없으며, 문자열 처리와 두-포인터 기법이 가장 적절하고 효율적인 "정석" 풀이 방법입니다. 현재 풀이는 이 정석 접근법을 따르고 있습니다.

### 4. 현재 풀이의 적절성과 개선 가능한 부분

*   **적절성:**
    *   코드의 로직은 정확하며 문제의 요구사항을 완벽하게 만족합니다.
    *   전처리 후 두-포인터 비교라는 핵심 아이디어를 잘 구현했습니다.
    *   가독성도 높은 편입니다.
*   **엣지 케이스 처리:**
    *   **빈 문자열 `""`:** `letters`가 `[]`가 되고 `left=0`, `right=-1`이 되어 `while left < right` 조건이 거짓이 됩니다. `True`를 반환하며 올바르게 처리됩니다. (빈 문자열은 팰린드롬으로 간주됨)
    *   **단일 문자열 `"a"`:** `letters`가 `['a']`가 되고 `left=0`, `right=0`이 되어 `while left < right` 조건이 거짓이 됩니다. `True`를 반환하며 올바르게 처리됩니다.
    *   **숫자 포함 `"racecar1"`:** `letters`가 `['r', 'a', 'c', 'e', 'c', 'a', 'r', '1']`가 되어 올바르게 비교합니다.
    *   **대소문자 혼합 `"Racecar"`:** `lower()` 덕분에 `['r', 'a', 'c', 'e', 'c', 'a', 'r']`로 변환되어 올바르게 비교합니다.
    *   **알파벳/숫자 외 문자 포함 `"A man, a plan, a canal: Panama"`:** `filter` 덕분에 `['a', 'm', 'a', 'n', 'a', 'p', 'l', 'a', 'n', 'a', 'c', 'a', 'n', 'a', 'l', 'p', 'a', 'n', 'a', 'm', 'a']`로 변환되어 올바르게 처리됩니다.

*   **개선 가능한 부분 (주로 공간 복잡도 최적화):**
    *   **`O(1)` 공간 복잡도 달성:** 위에서 언급했듯이, 추가적인 리스트 없이 원본 문자열 `s`에 두-포인터를 직접 적용하여 공간 복잡도를 `O(1)`로 만들 수 있습니다. 이 경우, 포인터가 가리키는 문자가 유효한 문자가 아닐 때 스킵하는 로직이 추가됩니다.

    ```python
    class Solution:
        def isPalindrome(self, s: str) -> bool:
            left, right = 0, len(s) - 1

            while left < right:
                # 왼쪽 포인터 이동: 유효한 문자(알파벳 또는 숫자)를 찾을 때까지
                while left < right and not s[left].isalnum():
                    left += 1
                # 오른쪽 포인터 이동: 유효한 문자(알파벳 또는 숫자)를 찾을 때까지
                while left < right and not s[right].isalnum():
                    right -= 1

                # 유효한 문자 쌍을 찾았으면 비교
                if s[left].lower() != s[right].lower():
                    return False

                # 다음 비교를 위해 포인터 이동
                left += 1
                right -= 1

            return True
    ```
    이 개선된 버전은 `letters` 리스트를 생성하지 않으므로 `O(1)` 공간을 사용합니다. 시간 복잡도는 여전히 `O(N)`이며, 각 문자를 최대 두 번(유효성 검사 및 비교) 확인합니다.

### 5. Python의 언어적 특성(Idiomatic Code) 준수 여부

*   **잘 준수된 부분:**
    *   `map`과 `filter`에 `lambda` 함수를 사용하는 것은 파이썬의 함수형 프로그래밍 스타일을 보여주는 idiomatic한 방식입니다.
    *   `x.lower()`, `x.isalpha()`, `x.isnumeric()`와 같은 문자열 메서드 활용도 적절합니다.
    *   두-포인터 로직(`left`, `right`) 구현도 파이썬스러운 방식입니다.
*   **개선할 수 있는 부분:**
    *   **`isalpha() or isnumeric()` 대신 `isalnum()` 사용:** `c.isalpha() or c.isnumeric()`는 `c.isalnum()`으로 더 간결하게 표현할 수 있습니다. 이는 문자열이 알파벳 또는 숫자인지 확인하는 전용 메서드입니다.
    *   **리스트 컴프리헨션(List Comprehension) 사용:** `map`과 `filter` 체인 대신 리스트 컴프리헨션을 사용하는 것이 파이썬에서는 더 읽기 쉽고 일반적인 패턴으로 간주될 때가 많습니다.

    ```python
    # 개선된 첫 번째 줄 (Idiomatic & 간결성)
    letters = [c.lower() for c in s if c.isalnum()]
    ```
    이 변경은 기능적으로 동일하지만, 많은 파이썬 개발자에게 더 익숙하고 직관적인 구문일 수 있습니다.
