# LeetCode No.1 Two Sum Code Review

시니어 알고리즘 코드 리뷰 요청에 따라 Swift로 작성된 LeetCode No.1 Two Sum 문제 풀이 코드를 리뷰합니다.

---

## 코드 리뷰: LeetCode No.1 Two Sum

### 1. 시간 복잡도 (Big-O 분석 및 최적화 가능성)

*   **`nums.enumerated()`**: `O(N)` - 배열의 모든 요소를 순회하며 인덱스와 값을 짝지어 시퀀스를 생성합니다.
*   **`.sorted { ... }`**: `O(N log N)` - 생성된 `(인덱스, 값)` 튜플 배열을 값(element)을 기준으로 정렬합니다. `N`은 `nums` 배열의 크기입니다.
*   **외부 `for` 루프**: `O(N)` - `pairs` 배열의 각 요소에 대해 한 번씩 반복합니다.
*   **내부 `while` 루프 (이진 탐색)**: `O(log N)` - 외부 루프의 각 반복마다 나머지 `pairs` 배열에서 이진 탐색을 수행합니다.
*   **총 시간 복잡도**: `O(N log N)` (정렬) + `O(N * log N)` (외부 루프 * 이진 탐색) = `O(N log N)`

**최적화 가능성**:
현재 풀이는 `O(N log N)`으로 효율적이지만, 이 문제의 "정석" 풀이로 알려진 해시 맵(Swift의 Dictionary)을 사용하는 방법은 `O(N)`의 시간 복잡도를 가집니다. 해시 맵 방법은 한 번의 순회로 각 숫자에 대해 `target - current_number`가 해시 맵에 존재하는지 확인하고, 없으면 현재 숫자를 해시 맵에 추가하는 방식입니다.

### 2. 공간 복잡도

*   **`pairs` 배열**: `O(N)` - `nums.enumerated().sorted()` 결과로 `(인덱스, 값)` 튜플을 `N`개 저장하는 새로운 배열이 생성됩니다.
*   **나머지 변수**: `left`, `right`, `mid`, `searching` 등은 모두 `O(1)` 공간을 사용합니다.
*   **총 공간 복잡도**: `O(N)`

### 3. 보편적인 "정석" 풀이 방법

`Two Sum` 문제의 보편적인 "정석" 풀이 방법은 다음과 같습니다:

1.  **해시 맵 (Dictionary) 사용**: `O(N)` 시간, `O(N)` 공간
    *   배열을 한 번 순회합니다.
    *   각 숫자 `num`에 대해 `target - num` (나머지 값)이 해시 맵에 존재하는지 확인합니다.
    *   존재한다면, 현재 숫자의 인덱스와 해시 맵에 저장된 나머지 값의 인덱스를 반환합니다.
    *   존재하지 않는다면, 현재 숫자 `num`과 그 인덱스를 해시 맵에 저장합니다.
    *   이 방법은 대부분의 경우 가장 효율적인 접근 방식으로 간주됩니다.

2.  **정렬 + 투 포인터**: `O(N log N)` 시간, `O(N)` 또는 `O(1)` 공간 (원본 인덱스 유지 여부에 따라 다름)
    *   배열을 정렬합니다. 이 과정에서 원본 인덱스를 유지해야 합니다 (현재 코드와 같이 `(인덱스, 값)` 튜플을 사용).
    *   정렬된 배열의 양 끝에 두 개의 포인터를 두고, 합이 `target`과 같은지 비교하며 포인터를 이동시킵니다.
    *   합이 `target`보다 작으면 왼쪽 포인터를 오른쪽으로, 크면 오른쪽 포인터를 왼쪽으로 이동합니다.

현재 풀이는 **정렬 + 이진 탐색** 방식입니다. 이 방식도 `O(N log N)`으로 정렬 + 투 포인터와 동일한 시간 복잡도를 가지며, brute-force(`O(N^2)`)보다는 훨씬 효율적입니다. 하지만 일반적으로는 해시 맵 풀이가 더 최적의 방법으로 여겨집니다.

### 4. 현재 풀이의 적절성과 개선 가능한 부분

*   **적절성**:
    *   이 풀이는 논리적으로 정확하며, `target`을 만족하는 두 숫자의 원래 인덱스를 성공적으로 찾아 반환합니다.
    *   `nums.enumerated()`를 사용하여 원본 인덱스를 보존하고, 정렬 후에도 이를 활용하는 방식은 적절합니다.
    *   이진 탐색을 통해 나머지 값을 찾는 방법론은 정렬된 배열에서 특정 값을 찾는 데 효과적입니다.
*   **개선 가능한 부분 (미미한 수준)**:
    *   **이진 탐색 `mid` 계산**: `mid = (left + right) / 2`는 `left`와 `right`가 매우 큰 경우 정수 오버플로우를 일으킬 가능성이 있습니다 (일반적인 LeetCode `N` 제약 조건에서는 드물지만, 좋은 습관). `mid = left + (right - left) / 2`가 더 안전합니다.
    *   **엣지 케이스**:
        *   `nums`가 비어있거나 한 개의 요소만 있는 경우: `pairs.count - 1`이 `0` 미만이 되므로 `for` 루프가 실행되지 않고 `[]`가 올바르게 반환됩니다.
        *   `target`을 만족하는 두 숫자가 없는 경우: 루프가 모두 실행된 후 `[]`가 반환됩니다. (문제 조건에서 항상 하나의 해답이 있다고 명시되어 있는 경우가 많지만, 안전하게 처리됩니다.)
        *   동일한 요소를 두 번 사용하지 않는 조건: `left = i + 1`을 통해 `pairs[i]` 자체는 이진 탐색 대상에서 제외되므로, 동일한 인덱스의 요소를 두 번 사용하지 않는다는 조건을 잘 만족합니다.

### 5. Swift의 언어적 특성(Idiomatic Code) 준수 여부

*   **`nums.enumerated()`**: Swift의 `enumerated()` 메서드를 사용하여 배열의 값과 인덱스를 동시에 얻는 것은 매우 관용적이고 적절합니다. 이 문제에서 원본 인덱스를 유지하는 데 필수적입니다.
*   **Trailing Closure (`sorted { ... }`)**: `sorted(by:)` 메서드에 클로저를 trailing closure 문법으로 전달하는 것은 Swift에서 매우 흔하고 가독성 높은 패턴입니다.
*   **변수명**: `left`, `right`, `mid`, `searching` 등은 알고리즘 맥락에서 매우 명확하고 이해하기 쉬운 변수명입니다.
*   **타입 추론**: 명시적인 타입 선언 없이도 Swift의 타입 추론 기능을 잘 활용하고 있습니다.
*   **상수 및 변수 선언 (`let`, `var`)**: 적절하게 사용되었습니다. `pairs`, `searching`은 `let`으로 선언하여 불변성을 유지하고, `left`, `right` 등은 `var`로 선언하여 변경 가능하게 한 점이 좋습니다.

**전반적인 평가**:
제공된 코드는 `O(N log N)` 시간 복잡도와 `O(N)` 공간 복잡도를 가지는 "정렬 후 이진 탐색" 방식을 사용하여 Two Sum 문제를 정확하고 효율적으로 해결합니다. Swift 언어의 관용적인 표현 (`enumerated()`, trailing closure)을 잘 활용하여 가독성도 뛰어납니다. 비록 `O(N)` 해시 맵 풀이가 시간 복잡도 면에서 더 최적의 솔루션으로 간주되지만, 현재 풀이도 충분히 훌륭한 알고리즘적 접근 방식이며 코드 품질 또한 높습니다.
