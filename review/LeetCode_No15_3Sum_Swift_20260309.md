# LeetCode No.15 3Sum Code Review

## LeetCode No.15 3Sum 풀이 코드 리뷰

안녕하세요, 시니어 알고리즘 코드 리뷰어입니다. Swift로 작성된 LeetCode No.15 3Sum 문제 풀이 코드를 아래 기준에 따라 리뷰하겠습니다.

---

### 1. 코드 개요

제공된 솔루션은 `threeSum` 함수가 주어지면, 배열 `nums`에서 세 숫자의 합이 0이 되는 모든 고유한 삼중항(triplet)을 찾아 반환합니다. 이 솔루션은 내부적으로 `twoSum` 헬퍼 함수를 사용하여 첫 번째 숫자를 고정한 후 나머지 두 숫자를 찾습니다. 중복 삼중항을 제거하기 위해 `Set<Triples>`를 사용합니다.

---

### 2. 시간 복잡도 (Big-O 분석 및 최적화 가능성)

**`twoSum` 함수:**
*   `nums` 배열(슬라이스)을 한 번 순회합니다: `O(N)` (여기서 `N`은 `nums` 슬라이스의 길이).
*   `occurence` 딕셔너리 접근 및 삽입 연산은 평균적으로 `O(1)`입니다.
*   **전체:** `O(N)`

**`threeSum` 함수:**
*   **외부 루프:** `nums` 배열을 `M`번 순회합니다 (여기서 `M`은 원본 `nums` 배열의 길이). `O(M)`
*   **`Array(nums[index...])`:** 각 반복마다 `nums`의 나머지 부분을 복사하여 새 배열을 만듭니다. 이 작업은 `O(M - index)` 시간이 소요되며, 외부 루프 내에서 반복되므로 총 `O(M^2)`의 복사 오버헤드가 발생합니다.
*   **`twoSum` 호출:** 각 `twoSum` 호출은 `O(M - index)`입니다. 외부 루프 내에서 반복되므로 총 `O(M^2)` 시간이 소요됩니다.
*   **`Set<Triples>` 삽입:** `twoSum`이 반환하는 `candidates` 배열은 최대 `O(M - index)` 개의 쌍을 포함할 수 있습니다. 각 `Triples` 객체의 `Hashable` 연산(해시 계산 및 비교)은 삼중항의 크기가 상수(3)이므로 `O(1)`로 간주할 수 있습니다. 따라서 모든 `candidates`를 삽입하는 데 평균 `O(M - index)` 시간이 소요되며, 이는 외부 루프 내에서 총 `O(M^2)`가 됩니다.
*   **최종 변환:** `Array(result).map { $0.parsed }`는 `K`개의 고유 삼중항이 있다면 `O(K)` 시간이 걸립니다. 최악의 경우 `K`는 `O(M^3)`가 될 수 있습니다.
*   **전체 시간 복잡도:** `Array` 슬라이스 복사 및 중첩된 루프 구조로 인해 지배적인 항은 **`O(M^3)`** 입니다.

**최적화 가능성:**
현재 `O(M^3)` 시간 복잡도는 3Sum 문제에 대해 최적화되지 않은 상태입니다. 일반적으로 최적화된 솔루션은 `O(M^2)`의 시간 복잡도를 가집니다. 가장 큰 개선점은 `Array(nums[index...])`와 `Set<Triples>`를 제거하고 정렬과 투 포인터(Two-Pointers) 접근 방식을 사용하는 것입니다.

---

### 3. 공간 복잡도

**`twoSum` 함수:**
*   `occurence` 딕셔너리: `nums` 슬라이스의 최대 `N`개 요소를 저장할 수 있으므로 `O(N)` 공간.
*   `result`: 최대 `N`개의 `[Int: Int]` 딕셔너리를 저장할 수 있으므로 `O(N)` 공간.
*   **전체:** `O(N)`

**`threeSum` 함수:**
*   `result`: `Set<Triples>`는 최악의 경우 `O(M^3)`개의 고유 삼중항을 저장할 수 있습니다 (예: `[-1, 0, 1, 2, ..., N]`과 같이 많은 조합이 가능한 경우). 각 `Triples` 객체는 크기가 3인 `[Int: Int]` 맵을 저장하므로, 총 **`O(M^3)`** 공간이 필요합니다.
*   `Array(nums[index...])`: 각 반복마다 새로운 배열을 생성하지만, 이는 임시적인 공간이며 동시에 `O(M)`을 초과하지 않습니다. 그러나 누적된 복사 작업은 시간 복잡도에 영향을 미칩니다.
*   `candidates`: `twoSum`의 결과로 최대 `O(M)`개의 `Triples`를 저장할 수 있습니다.
*   **전체 공간 복잡도:** `Set<Triples>`로 인해 **`O(M^3)`** 입니다.

### 4. 보편적인 "정석" 풀이 방법 (DP, 그래프 등 문제 유형에 맞는 접근법)

3Sum 문제를 위한 보편적인 "정석" 풀이 방법은 다음과 같습니다.

1.  **배열 정렬:** 먼저 입력 배열 `nums`를 오름차순으로 정렬합니다. `O(N log N)`
2.  **외부 루프 (첫 번째 숫자 고정):** `i`를 `0`부터 `N-3`까지 순회하며 첫 번째 숫자를 `nums[i]`로 고정합니다.
    *   **중복 건너뛰기:** 만약 `i > 0`이고 `nums[i] == nums[i-1]`이면, 이전 반복과 같은 삼중항을 생성하므로 현재 `i`를 건너뜁니다.
3.  **투 포인터 (나머지 두 숫자 찾기):** 각 `nums[i]`에 대해 `left = i + 1`과 `right = N - 1` 두 개의 포인터를 설정합니다.
    *   두 숫자의 합이 `target_two_sum = -nums[i]`가 되도록 `nums[left]`와 `nums[right]`를 찾습니다.
    *   `left < right`인 동안 반복합니다:
        *   `current_sum = nums[left] + nums[right]`를 계산합니다.
        *   `current_sum == target_two_sum` 이면:
            *   유효한 삼중항 `[nums[i], nums[left], nums[right]]`을 결과에 추가합니다.
            *   **중복 건너뛰기:** `left`를 증가시키면서 `nums[left] == nums[left-1]`인 동안 계속 증가시킵니다. 마찬가지로 `right`를 감소시키면서 `nums[right] == nums[right+1]`인 동안 계속 감소시킵니다.
            *   이후 `left++`와 `right--`를 수행하여 다음 쌍을 찾습니다.
        *   `current_sum < target_two_sum` 이면: `left++` (합을 키워야 함).
        *   `current_sum > target_two_sum` 이면: `right--` (합을 줄여야 함).
4.  결과: 고유한 삼중항들의 리스트.

**정석 풀이의 복잡도:**
*   **시간:** 정렬 `O(N log N)` + 투 포인터(`N`번의 외부 루프 * `N`번의 내부 루프) `O(N^2)`. 총 **`O(N^2)`**.
*   **공간:** `O(1)` (출력 배열 제외) 또는 정렬을 위해 새 배열을 만들 경우 `O(N)`.

### 5. 현재 풀이의 적절성과 개선 가능한 부분 (엣지 케이스 포함)

**적절성:**
*   `Set<Triples>`를 사용하여 고유한 삼중항을 처리하려는 시도는 옳습니다.
*   `threeSum`에서 `twoSum`을 호출할 때 `Array(nums[index...])`를 통해 현재 고정한 숫자 `number`를 제외한 나머지 부분을 탐색하도록 의도한 점은 맞습니다. (이는 `twoSum` 내부의 `if index == 0 { continue }` 로직 덕분입니다. `twoSum`이 받은 슬라이스의 첫 번째 요소는 `threeSum`의 `number`이기 때문입니다.)

**개선 가능한 부분 (주요 문제점):**

1.  **`twoSum` 함수의 정확성 문제 (매우 중요):**
    *   현재 `twoSum` 함수는 `(X, X)`와 같은 중복된 숫자로 구성된 쌍을 정확하게 찾지 못할 수 있습니다. 예를 들어 `nums = [0, 0]`, `target = 0`인 경우:
        *   `index=0, number=0`: `if index == 0 { continue }`에 의해 건너뜁니다.
        *   `index=1, number=0`: `finding = 0 - 0 = 0`. `occurence` 딕셔너리에 `0`이 없으므로 `if let found = occurence[finding]` 조건이 거짓이 되어 쌍을 찾지 못합니다.
        *   결과적으로 `[0, 0]` 쌍을 찾지 못합니다. 이는 `nums = [0, 0, 0]`과 같은 입력에 대해 `[0, 0, 0]` 삼중항을 놓치게 만듭니다. `twoSum`은 `occurence`에 이미 있는 숫자와 현재 숫자를 짝지어주기 때문에, `(X, X)` 쌍을 찾으려면 첫 번째 `X`가 `occurence`에 추가되어야 하고, 두 번째 `X`가 `target - X`로 발견되어야 합니다. 현재 로직에서는 `finding == number`인 경우, `number`가 `occurence`에 있을 때만 작동합니다.
    *   이는 `twoSum`의 `occurence` 딕셔너리 로직이 `[X, Y, X]` 형태에서 `(X,X)` 쌍을 찾지 못하는 문제입니다.

2.  **성능 문제 (시간 및 공간):**
    *   위에서 언급했듯이 `O(N^3)` 시간 복잡도와 `O(N^3)` 공간 복잡도는 매우 비효율적입니다. `Array(nums[index...])`를 통해 매번 새 배열을 복사하는 것은 큰 성능 저하를 초래합니다. `Set<Triples>`를 통한 중복 제거도 `O(N^3)`의 공간을 사용하게 만듭니다.

3.  **`twoSum`의 불필요한 복잡성 및 비표준 반환 타입:**
    *   `twoSum`이 `[[Int: Int]]` 형태의 빈도 맵을 반환하는 것은 `threeSum`에서 `Triples` 구조체로 변환하는 과정을 복잡하게 만듭니다. 일반적으로 `twoSum`은 발견된 값들의 쌍(`[[Int]]`)이나 인덱스(`[[Int]]`)를 반환합니다.

4.  **엣지 케이스:**
    *   `nums` 배열의 길이가 3 미만인 경우: `threeSum`은 빈 배열을 반환하며 올바르게 처리됩니다.
    *   `[0,0,0]`과 같은 모든 요소가 0인 경우: 위에서 언급했듯이 `twoSum`의 문제로 인해 `[0,0,0]` 삼중항을 찾지 못합니다.

**개선 방안:**
*   가장 큰 개선은 **정렬 + 투 포인터**의 `O(N^2)` 알고리즘으로 전환하는 것입니다. 이 방법은 시간 및 공간 효율성 면에서 월등하며, 중복 삼중항 처리도 더 효율적으로 수행할 수 있습니다.
*   `Triples` 구조체와 `Set` 기반의 중복 제거 로직은 정렬 + 투 포인터 방식에서는 필요하지 않습니다.

### 6. Swift의 언어적 특성(Idiomatic Code)을 잘 살렸는지 준수 여부

*   **구조체(`struct Triples: Hashable`) 활용:** `Triples` 구조체를 정의하고 `Hashable` 프로토콜을 채택하여 `Set`에 넣는 방식은 Swift의 값 타입과 프로토콜을 잘 활용한 예시입니다. 이는 Swift에서 커스텀 객체의 동등성 및 해싱을 다루는 일반적인 방법입니다.
*   **옵셔널 바인딩 (`if let found = occurence[finding]`):** 딕셔너리에서 값을 검색할 때 옵셔널 바인딩을 사용하는 것은 Swift의 안전한 코딩 스타일을 따릅니다.
*   **`enumerated()`:** `for (index, number) in nums.enumerated()` 패턴은 배열의 요소와 인덱스를 동시에 순회할 때 Swift에서 권장되는 방식입니다.
*   **`Array(nums[index...])`의 사용:** 이 부분은 Swift의 슬라이싱 기능을 활용한 것이지만, 매번 새로운 배열을 생성하는 것은 성능 최적화 관점에서 비idiomatic합니다. 성능이 중요한 경우, 가능한 한 슬라이스(`ArraySlice`)를 직접 사용하거나, 아예 복사를 피하는 다른 알고리즘을 고려해야 합니다. `twoSum` 함수가 `[Int]`를 파라미터로 받기 때문에 결국 복사가 일어날 수밖에 없는 구조입니다.
*   **`parsed` 속성:** `parsed` 연산 프로퍼티는 `[Int]` 배열을 반환하지만, 삼중항 내부 요소의 순서를 보장하지 않습니다. 문제에서 특정 순서를 요구하지 않는다면 괜찮지만, 일반적으로 출력은 정렬된 형태로 제공하는 것이 좋습니다 (`result.append(currentTriplet.sorted())`).

### 결론 및 종합 의견

제공된 솔루션은 Swift의 언어적 특성(구조체, 옵셔널 바인딩, `enumerated()`)을 일부 잘 활용하고 있으나, 알고리즘 자체의 효율성과 정확성 면에서 몇 가지 중요한 개선이 필요합니다.

*   **`twoSum` 함수의 `(X, X)` 쌍 처리 문제**로 인해 특정 엣지 케이스(예: `[0,0,0]`)에서 **정확성이 떨어집니다.**
*   **시간 복잡도 `O(N^3)`** 및 **공간 복잡도 `O(N^3)`**는 최적의 솔루션(`O(N^2)` 시간, `O(1)` 공간)에 비해 현저히 낮습니다. 특히 `Array` 슬라이스를 반복적으로 복사하는 것이 큰 성능 병목 지점입니다.

**권장 개선 방향:**
가장 권장되는 개선 방향은 배열을 **정렬하고 투 포인터 기법**을 사용하여 문제를 해결하는 것입니다. 이 접근 방식은 시간 및 공간 효율성 측면에서 훨씬 우수하며, 중복 삼중항 처리도 간결하게 수행할 수 있습니다. `twoSum` 헬퍼 함수를 재설계하거나 제거하고, `Triples` 구조체 및 `Set` 기반의 중복 제거 로직도 정렬된 배열을 직접 조작하는 방식으로 대체할 수 있습니다.
